# LLM-Powered Prompt Router (FastAPI)

A production-style Python service that classifies user intent with a lightweight LLM prompt, then routes the request to a specialized expert persona for a higher-quality response.

## Overview

This project implements a two-step routing pipeline:

1. `classify_intent(message)` performs a low-cost intent classification call.
2. `route_and_respond(message, intent_data)` selects a persona prompt and generates the final response.

Supported intents:

- `code`
- `data`
- `writing`
- `career`
- `unclear`

If the classifier output is malformed or confidence is too low, the service safely falls back to `unclear` and asks a clarification question.

## Intent Classification Design

The classifier prompt is intentionally strict:

- It limits output labels to `code`, `data`, `writing`, `career`, `unclear`.
- It requires JSON-only output with schema:

```json
{
  "intent": "string",
  "confidence": 0.0
}
```

Classifier robustness includes:

- Direct JSON parsing.
- Fallback extraction when extra text wraps JSON.
- Safe default on any parse/API error:

```json
{
  "intent": "unclear",
  "confidence": 0.0
}
```

## Prompt Routing Design

`route_and_respond` behavior:

1. Read `intent` and `confidence`.
2. Return clarification question if `intent == "unclear"`.
3. Return clarification question if confidence is below threshold (default `0.7`).
4. Otherwise, choose the mapped persona prompt and call the LLM for final response.

Clarification response:

`Could you clarify your request? Are you asking about coding, data analysis, writing improvement, or career advice?`

## API

### Health

- Method: `GET`
- Path: `/health`

Response:

```json
{ "status": "ok" }
```

### Chat

- Method: `POST`
- Path: `/chat`

Request:

```json
{
  "message": "how do i sort a list of objects in python?"
}
```

Response:

```json
{
  "intent": "code",
  "confidence": 0.93,
  "response": "..."
}
```

## Logging

Every `/chat` request appends one JSON object per line to `logs/route_log.jsonl`.

Each entry includes at least:

- `intent`
- `confidence`
- `user_message`
- `final_response`

Additional key:

- `timestamp` (UTC ISO-8601)

Example line:

```json
{"timestamp":"2026-03-08T09:00:00+00:00","intent":"code","confidence":0.95,"user_message":"how do i sort a list of objects in python?","final_response":"Use sorted(items, key=lambda x: x.field_name)..."}
```

## Using Groq (OpenAI-Compatible)

This service works with Groq through the OpenAI-compatible API by setting `OPENAI_BASE_URL`.

Example `.env`:

```env
OPENAI_API_KEY=gsk_xxxxxxxxx
OPENAI_MODEL=llama-3.3-70b-versatile
OPENAI_BASE_URL=https://api.groq.com/openai/v1
```

Notes:

- `OPENAI_API_KEY` is required; the app raises a clear startup/runtime error if missing.
- `OPENAI_BASE_URL` is optional. If it is empty, the client uses default OpenAI behavior.
- All chat-completions calls use the configured `OPENAI_MODEL`.

### Groq with Docker

1. Copy `.env.example` to `.env`.
2. Set the Groq values shown above.
3. Run `docker compose up --build`.
4. Call `POST /chat` and verify responses are generated by your Groq-selected model.
